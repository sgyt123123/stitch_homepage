## 当前状态概览

* 技术栈：React + Vite + TypeScript；单页滚动分区，未使用 react-router。

* 国际化：自定义 LanguageContext，文案来源于 `src/shared/lib/locales/{zh.ts,en.ts}`；支持中英切换与本地存储。

* 表单：`src/features/contact/components/ContactForm.tsx` 已有字段与 UI，但未实现提交、校验与后端/第三方对接。

* 产品：解决方案页的产品卡片数据来自 i18n 文案对象，图片在 `public/images/...`。

* 部署/运维：支持构建与预览；缺少 CI/CD、SEO 基础（sitemap/robots）、隐私与合规页、监控与分析。

## 上线所需核心功能（不做过度设计）

1. 内容与国际化文件外部化，便于非技术人员维护
2. 联系表单提交与收集方案（邮件通知 + 后台列表导出）
3. 产品数据驱动化并可扩展（新增产品无需改代码）
4. 基础 SEO 与分析（sitemap、robots、OG、Plausible）
5. 法务与合规（隐私政策、联系邮箱、Cookie 声明如需）
6. 部署与运维（Vercel 托管 + GitHub CI/CD + 环境变量）

## 双语与内容配置：最简且可由非技术人员维护

* 目标：非技术人员通过可视化后台编辑中英文文案与产品数据，无需接触代码。

* 方案选择：采用 Decap CMS（原 Netlify CMS）+ Git/GitHub 后端（文件驱动，零服务器，易上手）。

* 文件结构调整：

  * 新增 `public/content/i18n/zh.json` 与 `public/content/i18n/en.json` 存放全部页面文案（导航、首页、关于、解决方案、联系、页脚）。

  * 新增 `public/content/products.json` 管理产品列表，字段包含 `id/slug`, `name`, `short`, `long`, `image`, `localeFields`（如需要中英差异）。

  * 在运行时通过 `fetch` 读取 JSON，初始化 LanguageContext 的 `t` 对象。

* CMS 管理：

  * 新增 `public/admin/` 与 `config.yml`，将上述 JSON 文件注册为集合，提供表单式编辑与预览；使用 GitHub 登录，变更直接触发 PR/commit。

  * 给市场/运营提供 CMS URL 与简易操作手册（新增、修改、发布）。

* 权衡：相较接入重型 Headless CMS（Contentful、Strapi），文件型 CMS 成本最低，足够满足当前官网需求。

## 表单提交与收集：轻量稳妥

* 目标：上线即用、低维护、可邮件通知并导出数据。

* 方案选择：Formspree（或 Getform 同类）作为第三方收集端，无需自建后端。

* 实施要点：

  * 在 `ContactForm.tsx` 增加 `onSubmit`，用 `fetch` POST 到 Formspree 表单端点；成功后提示感谢，失败后提示重试。

  * 基础校验：使用原生约束与轻量正则（姓名/邮箱/电话必填，主题/留言格式），避免引入大型依赖。

  * 收集与通知：开启邮件通知、垃圾提交防护（honeypot 字段）与日后 CSV 导出；如需国内合规或数据主权，后续可切换 Supabase 存储方案。

## 产品扩展：数据驱动，新增零代码

* 目标：后期新增产品只需在 CMS 中添加一条数据，页面自动渲染。

* 数据模型：`products.json` 每项包含 `id/slug`, `name`, `descriptionShort`, `descriptionLong`, `image`, `enabled`, `order`；如中英文差异，按语言存放或使用 `translations` 字段。

* 页面逻辑：`SolutionsContent` 以当前语言过滤并排序渲染；图片从 `public/images/products/` 引用；CTA 链接可配置。

* 未来演进：如需详情页，再引入简易路由并根据 `slug` 生成详情；当前阶段保持单页，不引入路由以避免复杂度。

## SEO、分析与合规：上线基础配齐

* SEO 基础：添加 `sitemap.xml` 与 `robots.txt`；为各区段补充 `meta`/OG；语义化标签与可访问性（alt、aria、焦点状态）。

* 分析监控：集成轻量化 `Plausible`（或 Umami），仅需在 `index.html` 注入脚本与 `VITE_PLAUSIBLE_DOMAIN` 环境变量。

* 法务与合规：新增隐私政策页（或页脚链接至外部文档）、公司联系邮箱；如启用 Cookie/第三方追踪，提供 Cookie 声明。

## 部署与运维：简化且可持续

* 托管：Vercel（最佳与 Vite/React 兼容）；备选 Netlify。开启自动从 `main` 分支部署。

* CI/CD：GitHub Actions（`build` + `lint` + `test`）；CMS 提交触发预览与生产构建。

* 环境变量：新增 `.env` 使用 `VITE_*`（如 `VITE_FORM_ENDPOINT`, `VITE_PLAUSIBLE_DOMAIN`）；避免在代码中硬编码。

## 风险与取舍

* 文件型 CMS 依赖 GitHub 登录与仓库权限；如企业不便使用，可改为 Google Sheets 同步或轻量后端，但复杂度上升。

* 第三方表单服务有配额与地域访问差异；若面向国内用户，需要验证可用性或改为 Supabase 自建存储。
